# PHASE 12 — CI/CD

## Overview
The platform itself must be deployed securely and reliably, and it must also integrate into the CI/CD pipelines of the target applications it tests. This phase defines how the Autonomous Testing Platform is built, tested, and deployed, as well as how target applications trigger autonomous exploratory runs.

## Responsibilities
- Ensure the orchestration and worker layers are deployed without downtime.
- Gate target application deployments based on deterministic regression runs.
- Trigger long-running exploratory sessions asynchronously.

## Interfaces
- `Github Actions` / `GitLab CI` YAML configurations.

## Data Structures
```yaml
name: Deploy Platform
on:
  push:
    branches: [main]
jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps: ...
```

## Failure Modes
- CI Pipeline times out waiting for long-running exploratory tests to finish.
- Playwright browser cache misses causing massive npm install times.

## Recovery
- **Timeout**: Do not block the CI pipeline with exploratory tests. Exploratory tests run out-of-band (e.g., nightly). CI is only blocked by the *deterministic regressions* previously generated by the system.
- **Cache Miss**: Aggressive caching of `~/.cache/ms-playwright`.

## Tradeoffs
- **Containers vs Serverless**: The worker nodes require persistent, heavy browser dependencies (Playwright binaries). *Tradeoff*: We deploy workers as ECS Fargate containers rather than AWS Lambda, as Lambda's environment limitations and 15-minute timeout are incompatible with complex browser automation.

## Implementation Notes
- Use Turborepo remote caching to speed up monorepo builds.

## Future Evolution
- Implementing a custom GitHub App that comments directly on pull requests with screenshots of visual bugs discovered during the autonomous run.

## PIPELINE ARCHITECTURE

### Pipeline Stages
1. **Lint & Typecheck**: Enforces strict boundaries (ESLint).
2. **Unit Tests**: Validates agent prompt builders and schema logic.
3. **Integration Tests**: Spins up local Redis/Postgres via Testcontainers and runs a mock exploration.
4. **Build**: Compiles TypeScript to JS, builds Docker images.
5. **Deploy Orchestrator**: Rolling update to ECS.
6. **Deploy Workers**: Scale down old tasks, scale up new tasks on ECS.

### Caching & Parallelization
- `turbo` handles parallel execution of steps across packages.
- Docker layer caching is utilized to prevent redownloading Chrome binaries unless the Playwright version is bumped.

### Artifact Publishing
- Build artifacts (Docker images) are pushed to ECR/GHCR.
- NPM packages (like `shared-contracts`) can be published to an internal registry if external apps need to consume the types.

### Reporting & Quality Gates
- **Quality Gate**: The platform cannot deploy if test coverage of the *deterministic execution layer* falls below 90%. (AI agents require lower coverage due to their non-deterministic nature, testing relies on mocks).
- **Reporting**: JUnit XML reports are uploaded as CI artifacts.

### Deployment Strategy
- **Orchestrator**: Blue/Green deployment to ensure no messages are lost.
- **Workers**: Rolling update. Workers finish their current task before shutting down (SIGTERM graceful shutdown).

### Rollback
- If the Orchestrator fails to start (e.g., bad DB migration), the CI pipeline automatically rolls back the ECS service to the previous task definition. DB migrations must be strictly backward compatible (expand and contract pattern).
